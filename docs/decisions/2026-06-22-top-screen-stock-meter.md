# トップ画面は案C「在庫メーター型」を採用する

<!-- ファイル名は YYYY-MM-DD-kebab-title.md。同日複数は末尾に -2 等 -->

## ステータス
accepted

## コンテキスト / 背景
トップ（買うもの, SCR-030）はアプリの核。pending と next_order_date 到来を統合表示し、買ったらチェックして消える運用。
デザインモックで3案を比較した（案A チェックリスト型／案B カード型／案C 在庫メーター型）。
「アプリを開けば必ず最新の買うものが見える」を信頼性のベースにしており、開いた瞬間に**緊急度が直感で伝わる**ことが重要。
参照: デザインモック（references/v2.1）／[[design-system]] 7章。

## 決定
**案C「在庫メーター型」を採用**する。
- 消費サイクルの残量を**メーターバー**で可視化し、**緊急度で並べ替え**る。
- 「そろそろ切れるもの N件」、グループ「もう切れてる・切れる直前」、残り日数（例「あと2日」「予定を3日超過」）とサイクル（約45日）を併記。
- 銘柄横断（カテゴリ追跡）行も同じメーターで表現（「ワイン全体で約30日 · 前回 OSCO」）。
- 空状態（0件）は「今は買うものなし／ストックは十分そろっています」と**肯定的**に見せる。
- 案A・案Bはモックに参考残置（不採用）。

## 却下した代替案
- **案A チェックリスト型** → 却下。1タップで消す最短動線は良いが、緊急度・サイクルの情報量が乏しく「今どれくらいヤバいか」が伝わらない。
- **案B カード型（情報リッチ）** → 却下。商品画像と底値を前面に出し価格判断しながら買えるが、要素が重く一覧の動線が鈍る。価格判断は SCR-022 に委ねられる。

## トレードオフ / 結果
- 得たもの: 開いた瞬間に緊急度が直感で伝わる。残量バー＋緊急度ソートで「まず何を買うべきか」が一目で分かる。
- 失ったもの: メーターバーと残量計算の実装コストは案Aより高い。価格は前面に出ない（必要なら詳細/価格比較へ遷移）。
