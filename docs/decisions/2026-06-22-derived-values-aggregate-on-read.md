# 価格・底値・参考サイクルは専用カラムに持たず購入記録から都度集計する

<!-- ファイル名は YYYY-MM-DD-kebab-title.md。同日複数は末尾に -2 等 -->

## ステータス
accepted

## コンテキスト / 背景
solosto は「自分の過去の買値の中での相対評価」で買い時を測る（リアルタイムEC価格は扱わない）。
底値・平均・直近比などの価格指標と、参考用の自動サイクル値を、どこに保持するかを決める必要があった。
個人用MVPで `purchase_logs` が真実の購入記録として常に存在する前提。
参照: システム定義書_v2.md 2.7 / 2.4 / 5.1。

## 決定
派生値（底値・平均・参考自動サイクル）は **products に専用カラムを持たせず、`purchase_logs` から都度集計**する。
- 底値 = `MIN(unit_price)`、平均 = `AVG(unit_price)`、直近比 = 最新 `unit_price` との差。
- cycle_mode=auto の参考自動値も専用カラムを置かず購入記録から都度算出。
- ただし「実際に通知・リスト判定に使う確定値」（`next_order_date`、manual時の `per_unit_cycle_days`）は別。これは判定の入力なのでカラムに持つ。

## 却下した代替案
- **底値・平均を products にキャッシュカラムで持つ** → 却下。ログ削除・修正時に再計算を忘れると静かに不整合が残る。二重持ちの整合性管理コストが個人用MVPに見合わない。
- **プラットフォーム別に底値カラムを分けて持つ** → 却下。判断は全体底値のみで行う方針（platform は検索フィルタ用途に限定）。カラムが増え集計の意味も薄い。

## トレードオフ / 結果
- 得たもの: ログを消すだけで指標が常に正しくなる。「真実の源は購入記録ひとつ」に統一でき、整合性バグの温床を消せる。
- 失ったもの: 表示のたびに集計クエリが走る（個人用の小規模データでは無視できるコスト）。将来データが膨らみ遅くなったら、その時点で集計ビュー/マテビューを別ADRで検討する。
