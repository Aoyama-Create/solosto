# 生イベントは粒度非依存で記録し、集計の粒度は読み出し時に選ぶ

## 主張
一般に、「どの粒度で集計・追跡するか」が後から変わりうる/利用者ごとに異なる場合は、
**生イベントを最も細かい粒度（実際に起きたまま）で記録し、集計の粒度は読み出し時の GROUP BY で選ぶ**。
集計結果を保持カラムに固定すると、粒度を変える＝データを作り直す羽目になり、移行のたびに不整合が生まれる。
生イベントを粒度非依存に保てば、粒度の切り替えは「読み方を変えるだけ」になり、履歴を壊さず行き来できる。

## 根拠となった経験
- [[decisions/2026-06-22-tracking-scope-category-cycle]] で、追跡を商品単位⇄カテゴリ単位で切り替えられるようにした。サイクル・状態を専用カラムに持たず purchase_logs を GROUP BY（商品 or カテゴリ）で集計したため、scope切替が「集計単位の変更だけ」になり購入履歴は無傷で済んだ。
- [[decisions/2026-06-22-derived-values-aggregate-on-read]] と同根（派生値を持たない）。粒度の自由度という別角度から同じ「真実の源は生イベント」を支持する。

## 反例 / 例外
- [要確認] 集計が高頻度・高コストで実測がボトルネックになるなら、読み出し時集計をやめて粒度ごとのマテビュー等を持つ判断がありうる（その場合も生イベントは最細粒度で残す）。
- 切替時に付随する状態（solosto の pending）まで自動で粒度変換しようとすると複雑化する。割り切ってリセットする方が不整合ゼロで済むことがある。
