# 不確実な配信チャネルに依存せず、開けば真実が再計算される設計を最後の砦に置く

## 主張
一般に、配信（push通知・メール・webhook）が「届くとは限らない」チャネルである以上、
**ユーザーの正しい行動を配信の到達に依存させない**。配信は「気づくきっかけ」に格下げし、
利用者がアプリを開いた瞬間に状態を再計算して真実を提示する経路（プル型）を最後の砦に据える。
さらに、到達できなかった経路を検知してフォールバック告知する層を足し、フェイルセーフを多層化する。

## 根拠となった経験
- [[decisions/2026-06-22-pull-first-notification-reliability]] で、iOS Web Push の不安定さを仕様として受け入れ、プル型（起動時再計算）＋バッジ二重経路＋送信ログ/失効検知の3層にした。`sent`(201) が端末表示を保証しない問題も、再計算で吸収した。

## 反例 / 例外
- [要確認] 配信到達が法的・安全上クリティカルな領域（決済・医療アラート等）では、到達保証そのものへの投資（再送・複数チャネル・到達確認）が必要で、「開けば直る」だけでは不十分。
- 再計算が重い/状態が外部依存で揃わない場合、プル型を最後の砦にできないことがある。
